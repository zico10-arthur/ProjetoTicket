# DsCode Project Structure

This document maps every file under `.dscode/` and `management/` in a DsCode project. Use it to locate runtime state, planning documents, specs, and configuration.

---
## Direction of development

This project will be developed, maintained, and evolved by autonomous AI. Therefore, every technical, architectural, documentary, operational, and organizational decision must be optimized for autonomous AI execution across the entire software lifecycle.

When working on this project, always prioritize clarity, traceability, predictability, automation, testability, objective documentation, and low coupling. The code, architecture, processes, scripts, tests, commit patterns, error messages, configuration files, and documentation must be written in a way that allows an AI agent to understand, modify, validate, maintain, debug, and evolve the system with the lowest possible ambiguity and the least possible dependency on external context.

This directive applies to all phases of the project, including conception, planning, implementation, refactoring, testing, review, build, packaging, distribution, publishing, support, corrective maintenance, evolutionary maintenance, technical auditing, and post-production operation.

Whenever there is a choice between a solution that is more “elegant” but implicit and a solution that is clearer, more explicit, and easier for an AI agent to operate safely, choose the clearer and more explicit solution. The goal is to maximize the autonomy, safety, reliability, and efficiency of AI as the primary development agent for this software.

## `.dscode/` — Runtime State (per project)

### `settings.json`
User configuration file. Created manually or via `/setup`. Read at app startup by `src/settings.ts`.
Contains: LLM provider keys, model catalog overrides, generation parameters (temperature, top_p, max_tokens),
thinking budget, cache mode, web search toggle, permission scopes.
**Never committed to git** — contains API keys.

### `budget.md`
Accumulated cost summary for this project. Auto-generated by `/budget` command.
Tracks: total tokens consumed, estimated USD cost, breakdown by model.

### `memory/`
Conversation history storage. Managed by `src/common/turn-memory.ts`.

#### `memory/index.json`
Memory index. Maps session IDs to arrays of turn file paths. Schema: `Record<string, string[]>`.

#### `memory/turn_YYYYMMDDTHHMMSS_xxxxxxxx.dctz`
Compressed turn file (binary). Contains a single conversation turn: user message, assistant response,
tool calls, tool results, token usage. Used for context continuation and compaction.

#### `memory/turns/YYYY-MM/turn_YYYYMMDDTHHMMSS_xxxxxxxx.json.zst`
Zstd-compressed JSON turn file. Raw uncompressed format, used for archival and debugging.
Larger than `.dctz` but human-readable after decompression.

## `management/` — SDD (Specification-Driven Development)

This directory is the **canonical SDD root**. All planning, architecture, and spec documents live here.

### `roadmap.md`
**Master spec index.** Contains the spec table (columns: #, Name, Status, References) and dependency graphs.
Statuses: `planned`, `created`, `verified`, `implemented`, `audited`, `in-progress`, `proposed`, `discarded`.
When asked "what specs exist?", read this file first. It is the single source of truth for spec status.

### `vision.md`
Product vision document. Organized as V1, V2, ... V28 sections. Each section describes what a spec or feature version
delivers, the problem it solves, and which spec(s) implement it. References to ADRs and other specs.

### `adr.md`
Architecture Decision Records. Documents key architectural choices with context, rationale, and consequences.
Format: `ADR-XXX: <title>` followed by Status, Context, Decision, Consequences sections.

### `arch.md`
Architecture overview. Component diagram, data flow between modules, key abstractions.
Modules documented: session management, LLM providers, tools, TUI, memory, MCP, skills, settings.

### `lessons.md`
Implementation lessons learned. Patterns that worked, anti-patterns to avoid, gotchas.
Referenced by specs as `L1`, `L2`, etc.

### `budget.md`
High-level project budget tracking. Not per-session (see `.dscode/budget.md` for that).
Tracks overall project cost across all development sessions.

### `specs/`
Spec documents directory. Each subdirectory is named `<number>-<name>` (e.g., `120-explore-subagent`).

#### `specs/<N>-<name>/requirements.md`
Spec requirements. YAML frontmatter: `name`, `status`, `references`. Markdown body describes what the spec delivers,
acceptance criteria, and dependencies. This is the **spec contract**.

#### `specs/<N>-<name>/design.md`
Implementation design. Architecture decisions, data structures, algorithms, file list, API surface.
Written after requirements are approved.

#### `specs/<N>-<name>/task.md`
Task breakdown. Checklist of implementation steps. Each task is a small, verifiable unit of work.

#### `specs/<N>-<name>/prompt-implementacao.md` (rare)
Prompt template for AI-assisted implementation of this spec. Only present when spec was designed
for AI-driven execution.

## Quick Lookup

| If you need to... | Read... |
|---|---|
| Find all specs and their status | `management/roadmap.md` |
| Understand what a spec does | `management/specs/<N>-<name>/requirements.md` |
| Understand how a spec is built | `management/specs/<N>-<name>/design.md` |
| See implementation checklist | `management/specs/<N>-<name>/task.md` |
| Understand the product vision | `management/vision.md` |
| Find architectural decisions | `management/adr.md` |
| See the module architecture | `management/arch.md` |
| Check user settings | `.dscode/settings.json` |
| See session cost | `.dscode/budget.md` |
| Resume a previous session | `.dscode/memory/` (read `index.json` first) |
| Find lessons learned | `management/lessons.md` |
